## 一、分层架构

采用**四层架构**，自上而下为：UI 层、应用层、领域层、基础设施层。依赖方向严格单向：上层依赖下层，下层绝不感知上层。

```
┌─────────────────────────────────────────────────────────────┐
│                      UI 层 (Presentation)                     │
│  MainWindow, TransportBar, PlaylistPanel, CoverDisplay,     │
│  LibraryBrowser, SettingsDialog, MenuBarController,         │
│  LookAndFeel, ThemeManager                                  │
│  依赖：仅依赖应用层接口 (AppController & 数据模型)              │
├─────────────────────────────────────────────────────────────┤
│                    应用层 (Application)                       │
│  AppController (总控), PlayerController, LibraryController,  │
│  PlaylistController, SettingsManager                        │
│  职责：协调领域对象、管理应用生命周期、处理跨模块交互              │
│  依赖：仅依赖领域层接口                                       │
├─────────────────────────────────────────────────────────────┤
│                     领域层 (Domain)                           │
│  ┌─────────────┐  ┌──────────────┐  ┌──────────────────┐   │
│  │ AudioEngine  │  │ AudioDecoder  │  │ MusicLibrary     │   │
│  │ HogMode      │  │ FormatReader  │  │ Scanner          │   │
│  │ SampleRate   │  │ PCMBuffer     │  │ MetadataParser   │   │
│  │ IntegerMode  │  │               │  │ CoverArtLoader   │   │
│  └─────────────┘  └──────────────┘  └──────────────────┘   │
│  ┌─────────────┐  ┌──────────────┐                         │
│  │ Transport    │  │ Playlist     │                         │
│  │ StateMachine │  │ Engine       │                         │
│  └─────────────┘  └──────────────┘                         │
│  依赖：仅依赖基础设施层                                       │
├─────────────────────────────────────────────────────────────┤
│                   基础设施层 (Infrastructure)                  │
│  Logger (spdlog), ConfigStore (nlohmann/json),              │
│  Database (SQLite), PlatformUtils (CoreAudio helpers,       │
│  文件路径, 应用目录), ThreadPool (JUCE),                     │
│  AbstractFifo (JUCE), MessageThread (JUCE)                  │
│  依赖：仅依赖第三方库和系统 API                                │
└─────────────────────────────────────────────────────────────┘
```

### 1.1 各层职责与约束

| 层 | 职责 | 不可以做什么 |
|---|------|------------|
| **基础设施层** | 提供日志、配置、数据库、线程池、无锁队列等通用能力，封装第三方库和系统 API | 不能包含任何业务逻辑 |
| **领域层** | 实现音频播放、解码、音乐库、播放列表等核心业务逻辑，这是整个应用的"心脏" | 不能直接操作 UI 控件，不能直接访问文件系统（通过基础设施层的 PlatformUtils） |
| **应用层** | 协调领域对象完成用户用例（如"双击歌曲 → 播放"），管理应用生命周期 | 不能实现具体的音频处理逻辑，不能直接操作 UI 控件 |
| **UI 层** | 渲染界面、响应用户输入、展示数据 | 不能包含业务逻辑，不能直接访问数据库或文件系统 |

### 1.2 层间通信机制

| 通信方向 | 机制 | 示例 |
|---------|------|------|
| 上层 → 下层 | 直接方法调用 | `playerController.play(track_id)` 调用 `transport.play()` |
| 下层 → 上层 | **观察者模式（Listener）** | `Transport::Listener::onPositionChanged()` |
| 领域层 → UI 层 | `juce::MessageManager::callAsync()` + Listener | 音频引擎状态变化 → 异步通知 UI 刷新 |
| 音频线程 → 其他线程 | `juce::AbstractFifo` + 原子变量 | 音频电平数据 → UI VU 表 |
| 跨模块（同层）| 应用层协调，领域对象不直接互相依赖 | Transport 和 AudioEngine 通过 PlayerController 协调 |

## 二、模块划分

### 2.1 模块总览

```
BitPerfect/
├── src/
│   ├── infrastructure/     # 基础设施层（5 个模块）

### 2.2 基础设施层

| 模块 | 目录 | 职责 | JUCE 覆盖？ |
|------|------|------|-----------|
| **Logger** | `infrastructure/logger/` | spdlog 初始化、日志宏封装、日志级别控制 | 否（spdlog 替代 JUCE Logger） |
| **ConfigStore** | `infrastructure/config/` | JSON 配置文件读写、原子写入（tmp + rename）、默认值、变更通知 | 否（nlohmann/json） |
| **Database** | `infrastructure/database/` | SQLite 打开/关闭、SQL 执行封装、WAL 模式、事务、迁移 | 否（sqlite3） |
| **PlatformUtils** | `infrastructure/platform/` | 应用目录路径、文件操作、CoreAudio 设备枚举辅助 | 部分（JUCE File API + 自定义 CoreAudio 扩展） |
| **Threading** | `infrastructure/threading/` | 线程池创建、AbstractFifo 封装、异步任务提交 | 是（JUCE ThreadPool + AbstractFifo） |

### 2.3 领域层

| 模块 | 目录 | 职责 | JUCE 覆盖？ |
|------|------|------|-----------|
| **AudioEngine** | `domain/audio_engine/` | 设备管理、Hog Mode、采样率切换、Integer Mode、音频回调 | 部分（AudioDeviceManager + 自定义 CoreAudio 调用） |
| **AudioDecoder** | `domain/decoder/` | 音频文件解码、PCM 缓冲管理、预缓冲策略 | 部分（AudioFormatReader + 自定义缓冲管理） |
| **Transport** | `domain/transport/` | 播放状态机（Stop/Playing/Paused/Seeking）、播放模式（顺序/循环/随机） | **自行实现** |
| **MusicLibrary** | `domain/library/` | 目录扫描、元数据解析、增量更新、封面加载、数据库 CRUD | 部分（AudioFormatReader 元数据 + 自定义扫描逻辑 + SQLite） |
| **Playlist** | `domain/playlist/` | 播放列表 CRUD、M3U 导入导出、排序 | **自行实现** |
| **Metadata** | `domain/metadata/` | ID3v1/v2、Vorbis Comment、APE tag 解析、封面提取 | 部分（JUCE metadataValues + 自定义深度解析） |

### 2.4 应用层

| 模块 | 目录 | 职责 |
|------|------|------|
| **AppController** | `application/app_controller/` | 应用生命周期（启动/退出）、模块初始化/销毁顺序、单例管理 |
| **PlayerController** | `application/player_controller/` | 协调 Transport + AudioEngine + Decoder 完成播放用例 |
| **LibraryController** | `application/library_controller/` | 协调 MusicLibrary + Database 完成扫描、查询、封面获取 |
| **PlaylistController** | `application/playlist_controller/` | 协调 Playlist + Database 完成播放列表管理 |
| **SettingsManager** | `application/settings_manager/` | 配置读写门面、变更广播、热更新协调 |

### 2.5 UI 层

| 模块 | 目录 | 职责 |
|------|------|------|
| **MainWindow** | `ui/main_window/` | 主窗口容器、布局管理、菜单栏 |
| **TransportBar** | `ui/transport_bar/` | 播放/暂停/上下首按钮、进度条、音量滑块 |
| **NowPlaying** | `ui/now_playing/` | 当前曲目信息（标题、艺术家、专辑）、封面显示 |
| **PlaylistPanel** | `ui/playlist_panel/` | 播放列表视图（TableListBox）、拖拽排序 |
| **LibraryBrowser** | `ui/library_browser/` | 艺术家/专辑浏览树（TreeView）、搜索结果 |
| **SettingsDialog** | `ui/settings/` | 音频设备选择、Hog Mode/Integer Mode 开关、链路状态、缓冲区大小 |
| **ThemeManager** | `ui/theme/` | 浅色/深色主题、LookAndFeel 定制、跟随系统 |

### 3.2 基础设施层核心类

#### Logger（namespace 函数，非类）

```
文件：infrastructure/logger/Logger.h, Logger.cpp
职责：初始化 spdlog 异步 logger，提供 SPDLOG_INFO/DEBUG/ERROR 宏

// 初始化（应用启动时调用一次）
void initLogger(const std::string& log_dir);

// 设置日志级别（运行时通过设置面板调整）
void setLogLevel(spdlog::level::level_enum level);
```

**所属线程**：任何线程均可调用（spdlog 异步 logger 是无锁的）。

> ⚠️ 注意：音频实时线程中**禁止**调用 `SPDLOG_INFO` 等宏（即使是异步 logger，内部也可能涉及内存分配）。音频线程的错误通过原子变量标记，UI 线程定时检查后记录日志。

#### ConfigStore

```
文件：infrastructure/config/ConfigStore.h, ConfigStore.cpp
职责：读写 config.json，原子写入，变更通知

class ConfigStore {
public:
    // 加载配置文件（启动时调用）
    bool load(const juce::File& config_file);

    // 读取值（带默认值，key_path 格式如 "audio.sample_rate"）
    template<typename T>
    T getValue(const std::string& key_path, const T& default_value);

    // 写入值（不会立即落盘，需调用 save()）
    void setValue(const std::string& key_path, const nlohmann::json& value);

    // 原子保存到磁盘（写 .tmp → rename 覆盖 config.json，防断电丢失）
    bool save();

    // 获取底层 json 对象的只读引用（供批量读取场景）
    const nlohmann::json& getRoot() const;

private:
    nlohmann::json data_;      // 内存中的配置数据
    juce::File file_path_;      // 配置文件路径
    std::mutex mutex_;         // 保护读写，低频操作，mutex 可接受
};
```

**所属线程**：UI 线程写（用户操作设置），任意线程读（通过 mutex 保护）。

#### Database

```
文件：infrastructure/database/Database.h, Database.cpp
职责：SQLite 封装，提供参数化查询，WAL 模式

class Database {
public:
    bool open(const juce::File& db_file);
    void close();

    // 执行 SQL（INSERT/UPDATE/DELETE），返回受影响行数
    int execute(const std::string& sql,
                const std::vector<std::string>& params = {});

    // 查询（SELECT），每行结果通过回调处理
    void query(const std::string& sql,
               std::function<void(int col_count, char** values, char** col_names)> callback,
               const std::vector<std::string>& params = {});

    // 事务操作（批量写入时显著提升性能）
    void beginTransaction();
    void commit();
    void rollback();

    // 数据库版本迁移（首次创建或升级时调用）
    void migrate(int target_version);

    // 获取最后插入的 rowid
    int64 getLastInsertRowId() const;

private:
    sqlite3* db_ = nullptr;    // SQLite 数据库句柄
    std::mutex mutex_;         // SQLite 在 WAL 模式下支持多线程读，但写操作仍需协调
};
```

**所属线程**：后台线程（数据库操作耗时，且不涉及实时约束）。

### 3.3 领域层核心类

#### AudioEngine

```
文件：domain/audio_engine/AudioEngine.h, AudioEngine.cpp
职责：设备生命周期管理、Hog Mode、采样率切换、Integer Mode、音频回调分发

class AudioEngine : public juce::AudioIODeviceCallback {
public:
    // --- 生命周期 ---
    AudioEngine();
    ~AudioEngine();

    void initialize();   // 初始化 AudioDeviceManager，枚举所有音频设备
    void shutdown();     // 关闭设备，释放 Hog Mode

    // --- 设备管理 ---
    juce::StringArray getAvailableDevices();          // 获取可用设备列表
    void setCurrentDevice(const juce::String& device_name); // 切换设备
    juce::String getCurrentDeviceName() const;

    // --- Hog Mode ---
    bool isHogModeSupported() const;
    bool enableHogMode();
    void disableHogMode();
    bool isHogModeActive() const;

    // --- 采样率 ---
    bool setSampleRate(double sample_rate);
    double getCurrentSampleRate() const;
    juce::Array<double> getSupportedSampleRates() const;

    // --- Integer Mode ---
    bool isIntegerModeSupported() const;
    bool enableIntegerMode(int bit_depth);
    void disableIntegerMode();
    bool isIntegerModeActive() const;

    // --- 播放 ---
    void start();   // 启动音频设备，开始回调
    void stop();    // 停止音频设备
    bool isRunning() const;

    // --- 音频源注入（由 PlayerController 设置） ---
    // 设置音频数据源（AbstractFifo），音频线程从此读取 PCM 数据

    // --- Listener（状态变化通知） ---
    struct Listener {
        virtual ~Listener() = default;
        virtual void onDeviceChanged(const juce::String& device_name) {}
        virtual void onSampleRateChanged(double sample_rate) {}
        virtual void onHogModeChanged(bool active) {}
        virtual void onIntegerModeChanged(bool active) {}
        virtual void onError(const juce::String& message) {}
    };
    void addListener(Listener* listener);
    void removeListener(Listener* listener);

private:
    // juce::AudioIODeviceCallback 实现（运行在音频实时线程）
    void audioDeviceIOCallbackWithContext(
        const float* const* input_channel_data,
        int num_input_channels,
        float* const* output_channel_data,
        int num_output_channels,
        int num_samples,
        const juce::AudioIODeviceCallbackContext& context) override;

    void audioDeviceAboutToStart(juce::AudioIODevice* device) override;
    void audioDeviceStopped() override;

    // --- 成员变量 ---
    juce::AudioDeviceManager device_manager_;    // JUCE 设备管理器
    juce::ListenerList<Listener> listeners_;    // 观察者列表

    // 原子变量：音频线程安全读取，其他线程安全写入
    std::atomic<bool> is_running_{false};
    std::atomic<bool> hog_mode_active_{false};
    std::atomic<bool> integer_mode_active_{false};
    std::atomic<double> current_sample_rate_{44100.0};
    std::atomic<float> current_level_{0.0f};     // 当前电平值（供 VU 表使用）

    // 音频源（无锁读取）
    juce::AbstractFifo* audio_source_fifo_ = nullptr;
    int audio_source_channels_ = 2;
};
```

**所属线程**：

- 初始化/关闭：**UI 线程**
- 设备切换/采样率切换/Hog Mode：**UI 线程**（CoreAudio API 在 UI 线程调用）
- `audioDeviceIOCallback`：**系统音频线程（硬实时）**
- Listener 回调：通过 `MessageManager::callAsync` 投递到 **UI 线程**

**关键约束**：`audioDeviceIOCallback` 中绝对不能有锁、不能分配内存、不能做 I/O。只做三件事：
1. 从 `AbstractFifo` 读取解码好的 PCM 数据
2. `memcpy` 到 `output_channel_data`
3. 更新原子变量（播放位置、电平值）

#### AudioDecoder

```
文件：domain/decoder/AudioDecoder.h, AudioDecoder.cpp
职责：音频文件解码，管理预缓冲队列

class AudioDecoder {
public:
    // --- 文件信息 ---
    struct FileInfo {
        double sample_rate;
        int num_channels;
        int bit_depth;
        int64 total_samples;
        double duration_seconds;
        juce::String format_name;
    };

    // 打开文件并读取信息（不开始解码，快速返回）
    bool open(const juce::File& file, FileInfo& out_info);

    // 开始解码到内部缓冲区（在后台线程调用，阻塞式循环）
    void startDecoding();

    // 停止解码
    void stopDecoding();

    // 跳转到指定位置（sample 为单位，seek 操作）
    void seekTo(int64 samplePosition);

    // 获取无锁环形缓冲区（供音频线程读取 PCM 数据）
    juce::AbstractFifo& getFifo();

    // 是否已解码完成（文件末尾）
    bool isDecodingComplete() const;

    // 获取已解码到的位置
    int64 getDecodedPosition() const;

    // --- Listener ---
    struct Listener {
        virtual ~Listener() = default;
        virtual void onDecodingStarted() {}
        virtual void onDecodingComplete() {}
        virtual void onDecodingError(const juce::String& message) {}
    };
    void addListener(Listener* listener);
    void removeListener(Listener* listener);

private:
    // 后台线程执行的解码循环（由 ThreadPool 调度）
    void decodingLoop();

    std::unique_ptr<juce::AudioFormatReader> reader_;    // JUCE 格式读取器
    std::unique_ptr<juce::AudioFormatReaderSource> source_; // JUCE 音频源

    // AbstractFifo: 无锁环形缓冲区
    // 容量 = sample_rate * channels * sizeof(float) * 0.5 秒 ≈ 172KB（44.1kHz立体声）
    // 生产者（解码线程）写入，消费者（音频线程）读取
    juce::AbstractFifo fifo_{1024 * 512};  // 约 0.5 秒缓存

    // PCM 数据缓冲区
    juce::AudioBuffer<float> decode_buffer_;  // 解码线程写入
    juce::AudioBuffer<float> ready_buffer_;   // 准备交给音频线程（通过 Fifo 索引管理）

    // 原子变量：跨线程状态同步
    std::atomic<bool> running_{false};
    std::atomic<bool> decoding_complete_{false};
    std::atomic<int64> current_position_{0};  // 当前解码位置（sample 为单位）

    juce::ListenerList<Listener> listeners_;
};
```

**所属线程**：
- `open()` / `startDecoding()` / `stopDecoding()` / `seekTo()`：**后台线程**（由 PlayerController 通过 ThreadPool 调度）
- `getFifo()`：**音频实时线程**读取（AbstractFifo 无锁安全）
- Listener 回调：**UI 线程**（通过 MessageManager 转发）

#### Transport

```
文件：domain/transport/Transport.h, Transport.cpp
职责：播放状态机、播放模式管理、播放位置追踪

class Transport {
public:
    // --- 播放状态 ---
    enum class State { Stopped, Playing, Paused, Seeking };

    // --- 播放模式 ---
    enum class PlayMode { Sequential, RepeatOne, RepeatAll, Shuffle };

    // --- 播放命令 ---
    void play();
    void pause();
    void stop();
    void togglePlayPause();  // 播放 ↔ 暂停切换
    void next();
    void previous();
    void seekTo(double position_seconds);

    // --- 状态查询 ---
    State getState() const;
    PlayMode getPlayMode() const;
    double getCurrentPosition() const;   // 当前播放位置（秒）
    double getTotalDuration() const;     // 当前曲目总时长（秒）
    void setTotalDuration(double duration_seconds);

    // --- 播放模式切换 ---
    void setPlayMode(PlayMode mode);
    void cyclePlayMode();  // 顺序 → 单曲循环 → 列表循环 → 随机 → 顺序

    // --- Listener ---
    struct Listener {
        virtual ~Listener() = default;
        virtual void onStateChanged(State new_state) {}

## 四、线程模型

### 4.1 三类线程及约束

| 线程 | 优先级 | 约束 | 职责 |
|------|-------|------|------|
| **音频实时线程** | 系统最高（macOS `kAudioWorkInterval`） | **硬实时**：不能锁、不能分配内存、不能 I/O、不能调用系统 API | 从 AbstractFifo 读取 PCM → memcpy 到输出缓冲 → 更新原子位置 |
| **UI 主线程** | 正常 | 不能阻塞超过 16ms（否则掉帧），不能做 CPU 密集计算 | 渲染界面、响应用户输入、调用领域层控制命令、接收 Listener 通知 |
| **后台线程池** | 低 | 可以被抢占 | 音频解码、文件扫描、数据库写入、M3U 解析 |

### 4.2 线程间通信矩阵

```
音频线程 ──→ UI 线程：
  ├─ 播放位置 (std::atomic<double>，音频写，UI 定时读)
  ├─ 电平值 (std::atomic<float>，音频写，UI 定时读)
  └─ 错误标志 (std::atomic<bool>，UI 100ms 定时器检查)

UI 线程 ──→ 后台线程：
  └─ ThreadPool::addJob(lambda) 提交解码/扫描任务

后台线程 ──→ 音频线程：
  └─ AbstractFifo (解码器写入 PCM，音频回调读取)
     绝对不直接通信，只通过 AbstractFifo 内存中转

后台线程 ──→ UI 线程：
  ├─ MessageManager::callAsync() 投递结果
  └─ Listener 回调（内部使用 MessageManager 保证在 UI 线程执行）

UI 线程 ──→ 音频线程：
  ├─ std::atomic 变量（播放/暂停标志、seek 位置、音量）
  └─ AbstractFifo 控制命令（极少使用）
```

### 4.3 音频实时线程安全守则

在 `audioDeviceIOCallback` 中，以下行为是**严格禁止**的：

```cpp
void AudioEngine::audioDeviceIOCallbackWithContext(...) {
    // ✅ 可以做的事：
    //   1. memcpy（复制数据到输出 buffer）
    //   2. 原子变量读写（std::atomic）
    //   3. AbstractFifo 读写（无锁环形缓冲）
    //   4. 简单算术运算

    // ❌ 绝对不能做的事：
    //   1. new / delete / malloc / free（内存分配）
    //   2. 任何锁（std::mutex, juce::CriticalSection）
    //   3. 文件 I/O（fopen, fread 等）
    //   4. 系统 API 调用（CoreAudio setProperty 等——会导致死锁或优先级反转）
    //   5. spdlog 日志（即使是异步的，内部可能涉及分配）
    //   6. JUCE 消息发送（MessageManager::callAsync）
}
```

## 五、数据流

### 5.1 音频数据流（文件 → DAC）

音频数据流有两条路径，根据是否启用 Integer Mode 自动选择：

#### 路径 A：Float 模式（默认，P0 实现）

绝大多数场景走此路径。源文件无论什么位深，统一解码为 Float32，CoreAudio 原生消费，零转换。

```
┌──────────┐     ┌─────────────────┐     ┌──────────────┐
│ 磁盘文件  │ ──→ │ AudioDecoder     │ ──→ │ AbstractFifo │
│ .flac    │     │ 后台线程读取文件   │     │ 无锁环形缓冲   │
│ .wav ... │     │ AudioFormatReader │     │ (生产者-消费者) │
└──────────┘     │ 解码为 PCM float  │     └──────┬───────┘
                 └─────────────────┘            │
                                                 │ 音频实时线程读取
                                        ┌────────▼───────┐
                                        │ AudioEngine     │
                                        │ audioDevice     │
                                        │ IOCallback      │
                                        │ memcpy → output │
                                        └────────┬────────┘
                                                 │ Float32，零转换
                                        ┌────────▼───────┐
                                        │ CoreAudio HAL   │
                                        │ (Hog Mode)      │
                                        │ 标准 Float32 格式│
                                        └────────┬────────┘
                                                 │
                                        ┌────────▼───────┐
                                        │ USB DAC         │
                                        │ 物理设备         │
                                        └────────────────┘
```

#### 路径 B：Integer Mode（P0 实现）

启用 Integer Mode 后走此路径。解码器保留源文件的原始整数位深（Int16/Int24/Int32），
通过 CoreAudio 的 `kAudioStreamPropertyPhysicalFormat` 直接输出整数裸数据到 DAC，
**全程不经过 float 转换**，实现真正的 bit-perfect。

```
┌──────────┐     ┌─────────────────┐     ┌──────────────┐
│ 磁盘文件  │ ──→ │ AudioDecoder     │ ──→ │ AbstractFifo │
│ .flac    │     │ 后台线程读取文件   │     │ 无锁环形缓冲   │
│ .wav ... │     │ AudioFormatReader │     │ (生产者-消费者) │
└──────────┘     │ 解码为 PCM int   │     └──────┬───────┘
                 │ (保留原始位深)    │            │
                 │ Int16/Int24/Int32│            │ 音频实时线程读取
                 └─────────────────┘     ┌───────▼───────┐
                                         │ AudioEngine     │
                                         │ audioDevice     │
                                         │ IOCallback      │
                                         │ memcpy → output │
                                         └────────┬────────┘
                                                  │ 原始整数位深，零转换
                                         ┌────────▼──────────────┐
                                         │ CoreAudio HAL          │
                                         │ (Hog Mode)             │
                                         │ kAudioStreamProperty   │
                                         │ PhysicalFormat = IntXX │
                                         └────────┬──────────────┘
                                                  │
                                         ┌────────▼───────┐
                                         │ USB DAC         │
                                         │ 原生整数格式接收  │
                                         │ (bit-perfect)   │
                                         └────────────────┘
```

**两条路径的选择逻辑**（P0 实现，CLI 测试程序依次验证两条路径）：

```
AudioEngine::start()
      │
      ├── Integer Mode 已启用？
      │       │
      │       ├── 是 → 检查 DAC 是否支持源文件位深？
      │       │       │
      │       │       ├── 支持 → 走路径 B（PCM int 直出）
      │       │       └── 不支持 → 走路径 A（静默回退到 float）
      │       │
      │       └── 否 → 走路径 A（PCM float）
```

**关键设计点**：
1. **预缓冲**：解码器在后台线程提前解码 0.5 秒的 PCM 数据放入 AbstractFifo
2. **无锁交付**：音频线程只从 AbstractFifo 读取，Fifo 空时输出静音（防爆音）
3. **零格式转换**：两条路径都在音频回调中直接 memcpy，不做任何格式转换
   - 路径 A：Float32 → CoreAudio Float32，直接拷贝
   - 路径 B：Int16/Int24/Int32 → CoreAudio PhysicalFormat IntXX，直接拷贝
4. **解码线程承担转换成本**：所有格式转换（如源文件 16-bit int → Float32）都在后台解码线程完成，音频实时线程只做 memcpy
5. **静默回退**：Integer Mode 不可用时自动回退到 Float 路径，不影响正常播放（P0）

## 七、关键设计决策

### 7.1 为什么选择四层架构而非 MVC

传统 MVC 的"Model"层会混合基础设施（数据库/配置/日志）和领域逻辑（播放引擎/解码/状态机），导致：

1. **变更原因混杂**：数据库切换（基础设施）和播放逻辑修改（业务）会互相影响
2. **难以单独测试**：测试播放逻辑时必须连带数据库/文件系统
3. **违反单一职责**：基础设施因平台变化，领域因需求变化，UI 因设计变化

四层架构确保每层只有一个变更原因，且依赖方向严格单向。

### 7.2 音频缓冲区设计策略

**核心方案：AbstractFifo 三级流水线**

```
解码线程(后台)          AbstractFifo(无锁)         音频线程(硬实时)
     │                      │                         │
     ├─ 解码一帧             │                         │
     ├─ 写入 Fifo ──────────→│                         │
     │                      ├─ 缓冲等待               ├─ 读取 Fifo
     │                      │                        ├─ memcpy → output
     │                      ├─ 缓冲中                 │
     ├─ 解码下一帧           │                         │
     ├─ 写入 Fifo ──────────→│                         │
     │                      │                        ├─ 读取 Fifo
```

**缓冲区容量计算**：
```
Fifo 容量 = sample_rate × channels × sizeof(float) × bufferSeconds
        = 44100 × 2 × 4 × 0.5
        = 176,400 字节 ≈ 172 KB
```

0.5 秒缓冲足够应对临时线程调度延迟，内存占用极小，延迟可接受（< 500ms）。

**野缓冲保护**：Fifo 为空时输出零值（静音），防止输出未初始化内存。

### 7.3 播放列表 vs 音乐库的关系

两者独立，通过 TrackInfo 关联：

```
MusicLibrary (数据库)
  │  存储所有扫描到的曲目
  │  提供搜索、浏览、分类
  │  不关心"当前播放什么"
  │
  └──→ TrackInfo (数据载体) ──→ Playlist
                                  │  用户创建的播放列表
                                  │  当前播放队列
                                  │  引用 MusicLibrary 中的 db_id
                                  │  不存储音频数据
```

好处：
- 播放列表只存储 track_id 引用，不重复存储元数据
- 删除音乐库中文件 → 播放列表标记"文件丢失"，不崩溃
- M3U 导入时按文件路径查找已有记录，避免重复扫描

### 7.4 配置热更新机制

```
用户修改设置
  │
  ▼
SettingsManager::setValue(key, value)
  │
  ├── 1. 更新 ConfigStore 内存中的 json
  ├── 2. 原子写入磁盘（写 .tmp → rename 覆盖 config.json）
  ├── 3. 根据 key 路由通知对应模块：
  │      "audio.device"     → AudioEngine::setCurrentDevice()
  │      "audio.hog_mode"   → AudioEngine::enableHogMode()
  │      "audio.sample_rate"→ AudioEngine::setSampleRate()
  │      "ui.theme"         → ThemeManager::applyTheme()
  │      "library.folders"  → LibraryController::setMusicFolders()
  │
  └── 4. 通知 UI Listener（如有需要刷新的控件）
```

不立即生效的设置（如缓冲区大小）标记为"需要重启音频设备"，UI 给出提示。

### 7.5 错误处理策略

分层错误处理，每层将错误转换为本层的语义：

| 层 | 错误来源 | 处理方式 | 示例 |
|---|---------|---------|------|
| **基础设施层** | 文件不存在、数据库损坏 | 记录日志 + 返回错误码 | `Database::open()` 返回 `false` |
| **领域层** | 格式不支持、DAC 断开 | Listener 通知上层 + 尝试恢复 | `AudioDecoder::Listener::onDecodingError()` |
| **应用层** | 操作失败 | 记录日志 + 通知 UI + 降级处理 | 播放失败 → 跳过当前曲目，播下一首 |
| **UI 层** | 用户可见的错误 | 弹窗 / Toast / 状态栏提示 | "文件格式不支持" → 状态栏红色提示 |

**音频实时线程的特殊处理**：
- 音频回调中发生的错误**不能抛异常、不能调 Listener、不能写日志**
- 只能：设置原子错误标志 + 输出静音
- UI 线程通过 100ms 定时器检查原子错误标志，做出响应（如弹出"DAC 设备断开"提示）

## 九、数据库 Schema（P2 阶段创建）

```sql
-- 曲目表（核心数据表）
CREATE TABLE IF NOT EXISTS tracks (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    file_path     TEXT NOT NULL UNIQUE,     -- 文件绝对路径，唯一索引
    title         TEXT NOT NULL DEFAULT 'Unknown',
    artist        TEXT NOT NULL DEFAULT 'Unknown',
    album         TEXT NOT NULL DEFAULT 'Unknown',
    albuartist_  TEXT NOT NULL DEFAULT '',  -- 专辑艺术家（合辑场景与 artist 不同）
    track_number  INTEGER DEFAULT 0,
    disc_number   INTEGER DEFAULT 0,
    year          INTEGER DEFAULT 0,
    genre         TEXT DEFAULT '',
    duration      REAL DEFAULT 0.0,         -- 时长（秒）
    sample_rate   REAL DEFAULT 44100.0,     -- 采样率（Hz）
    bit_depth     INTEGER DEFAULT 16,
    channels      INTEGER DEFAULT 2,
    format        TEXT DEFAULT '',           -- 文件格式："FLAC", "WAV" 等
    file_size     INTEGER DEFAULT 0,        -- 文件大小（字节）
    has_cover     INTEGER DEFAULT 0,        -- 是否有内嵌封面（0/1）
    last_modified TEXT DEFAULT '',           -- 文件修改时间（ISO 8601），增量扫描依据
    date_added    TEXT DEFAULT (datetime('now'))  -- 首次入库时间
);

-- 索引（加速查询）
CREATE INDEX IF NOT EXISTS idx_tracks_artist ON tracks(artist);
CREATE INDEX IF NOT EXISTS idx_tracks_album ON tracks(album);
CREATE INDEX IF NOT EXISTS idx_tracks_title ON tracks(title);
CREATE INDEX IF NOT EXISTS idx_tracks_file_path ON tracks(file_path);

-- FTS5 全文搜索（标题 + 艺术家 + 专辑，支持中文和英文混合搜索）
CREATE VIRTUAL TABLE IF NOT EXISTS tracks_fts USING fts5(
    title,
    artist,
    album,
    content='tracks',      -- 关联到 tracks 表
    content_rowid='id'     -- 使用 id 作为 rowid
);

-- 播放列表表
CREATE TABLE IF NOT EXISTS playlists (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    name         TEXT NOT NULL,              -- 播放列表名称
    description  TEXT DEFAULT '',            -- 描述（可选）
    date_created TEXT DEFAULT (datetime('now')),
    date_modified TEXT DEFAULT (datetime('now'))
);

-- 播放列表-曲目关联表（多对多）
CREATE TABLE IF NOT EXISTS playlist_tracks (
    playlist_id INTEGER NOT NULL,
    track_id    INTEGER NOT NULL,
    position    INTEGER NOT NULL,           -- 排序位置（从 0 开始）
    date_added  TEXT DEFAULT (datetime('now')),
    PRIMARY KEY (playlist_id, track_id),
    FOREIGN KEY (playlist_id) REFERENCES playlists(id) ON DELETE CASCADE,
    FOREIGN KEY (track_id) REFERENCES tracks(id) ON DELETE CASCADE
);

-- 数据库版本管理（Migration 机制）
CREATE TABLE IF NOT EXISTS schema_version (
    version INTEGER NOT NULL
);

-- 初始化版本号（首次创建时插入）
INSERT INTO schema_version (version) VALUES (1);
```
