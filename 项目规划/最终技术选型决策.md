# BitPerfect 最终技术选型决策

> 本文档基于 13 份技术选型方案对比文档，结合项目定位和开发者偏好，做出最终技术选型决策。
>
> 决策日期：2026-07-07

---

## 一、决策原则

以下 5 条原则是本次选型的核心依据，优先级从高到低排列：

| 优先级 | 原则 | 说明 |
|:---:|------|------|
| 1 | **跨平台优先** | 应用定位是跨平台（macOS / Windows / Linux），所有技术选型必须支持跨平台，或至少有清晰的跨平台迁移路径 |
| 2 | **低学习成本** | 开发者是 C++ 和音频领域新手，技术和框架要容易学习和快速上手 |
| 3 | **统一跨平台 UI** | UI 风格不必贴合系统原生，只要简约好看，保持跨平台风格统一 |
| 4 | **成熟活跃的社区** | 技术要有成熟的应用案例，社区多人且活跃，官方文档多且详细，报错好找答案 |
| 5 | **开源友好** | 项目开源、个人使用、不考虑商业化，许可证需兼容开源分发 |

## 二、核心决策：选择 JUCE 全栈方案

在综合评估 13 个技术维度后，**核心决策是采用 JUCE 作为主体框架**，覆盖音频引擎、音频解码、UI 框架、多线程四大模块。其余模块（构建、日志、测试、配置、数据库等）选用各自领域的最佳独立方案。

### 为什么是 JUCE？

从新手视角看，构建一个跨平台音频播放器面临的最大挑战不是单个技术点，而是**技术栈碎片化**——你需要自己把「音频 I/O → 解码 → DSP → UI → 多线程」这 5 层粘合在一起。每多一层独立库，就多一份集成调试的心智负担。

JUCE 的一站式方案直接消除了这个粘合层：

| 新手面临的挑战 | 不用 JUCE 的代价 | JUCE 的解法 |
|---------------|-----------------|------------|
| 音频 I/O ↔ 解码 ↔ UI 的数据流怎么设计？ | 自己设计线程模型、缓冲队列、回调链 | JUCE 的 AudioSource → AudioIODeviceCallback 已经设计好 |
| 音频解码后怎么传给播放引擎？ | miniaudio 的 buffer 格式和 dr_libs 的 buffer 格式不一致，需要转换层 | AudioFormatReader 直接读到 AudioBuffer，零转换 |
| UI 上的 VU 表怎么拿到音频数据？ | 从音频实时线程通过无锁队列传到 UI 线程 | JUCE 的 AbstractFifo 专为此场景设计 |
| 多平台设备管理怎么统一？ | macOS 调 CoreAudio，Windows 调 WASAPI，Linux 调 ALSA，三套代码 | AudioDeviceManager 一套 API，自动选择后端 |
| 出问题时去哪找答案？ | 分散在 miniaudio / dr_libs / Qt / spdlog 各自社区 | JUCE Forum 一个地方，且音频播放器是最常见的 JUCE 项目类型 |

**最重要的理由**：JUCE 拥有音频开发领域**最好的文档和教程**。对于新手来说，好文档比简单 API 更重要——JUCE 的官方教程手把手教你构建一个完整的音频应用，而 miniaudio + dr_libs + Qt 的组合虽然每个单独看都很简单，但没人告诉你它们怎么拼在一起。

### GPLv3 许可证说明

JUCE 采用 GPLv3 许可证（或付费商业许可）。**GPLv3 与本项目完全兼容**：
- 项目是开源的（GitHub 公开仓库）
- 项目是个人使用、非商业化
- GPLv3 要求衍生作品也以 GPL 发布，这符合项目开源定位

### JUCE 的缺点认知

| 缺点 | 影响评估 | 应对策略 |
|------|---------|---------|
| 框架体量大（编译后数十 MB） | 可接受，现代桌面应用常见体积 | 仅启用需要的模块（juce_audio_devices / juce_audio_formats / juce_gui_basics / juce_dsp） |
| UI 非原生控件 | **与原则 3 一致**，本项目不需要原生外观 | 使用 JUCE LookAndFeel 定制简约现代风格，跨平台外观统一 |
| 大版本升级有 breaking changes | 开发期影响小 | 锁定 JUCE 大版本（如 JUCE 8），不频繁升级 |
| DSD 格式不支持 | 暂不处理 | DSD 用户群体过于小众，待应用主体开发完毕后视情况再考虑 |

## 三、各维度最终决策

### 3.1 IDE / 编辑器

**决策：VS Code（主力编辑器）+ Xcode（macOS 调试/性能分析辅助）**

| 维度 | 评估 |
|------|------|
| VS Code 角色 | 日常编码主力，跨平台一致体验。安装 C/C++（clangd）、CMake Tools 插件 |
| Xcode 角色 | macOS 上调试（LLDB GUI）、性能分析（Instruments 是必需工具，无法替代）、打包签名 |
| 为什么不选 CLion | 收费（$99+/年），且对 Swift/ObjC 支持极弱。本项目 macOS 阶段仍需 Xcode 辅助 |

> 参考文档：[开发工具链方案对比.md](技术选型参考对比/开发工具链方案对比.md)

---

### 3.2 构建系统

**决策：CMake**

| 维度 | 评估 |
|------|------|
| 跨平台 | ✅ macOS / Windows / Linux 统一使用 CMakeLists.txt |
| 社区生态 | C++ 生态事实标准，几乎所有 C++ 库都支持 CMake |
| VS Code 集成 | CMake Tools 插件提供一键 Configure / Build / Debug |
| JUCE 兼容 | JUCE 原生支持 CMake（通过 `juce_add_plugin` / `juce_add_gui_app` 等 CMake 函数） |
| 学习曲线 | 语法有些怪异，但这是绕不开的行业标准。配置模板写好后就很少改动 |

**为什么不选 xmake？** xmake 语法更简洁、中文文档更好，但 JUCE 的 CMake 集成更成熟。JUCE + CMake 是官方推荐组合。

> 参考文档：[构建系统方案对比.md](技术选型参考对比/构建系统方案对比.md)

---

### 3.3 音频引擎

**决策：JUCE AudioDeviceManager**

JUCE 的 `AudioDeviceManager` 负责：
- 跨平台音频设备枚举和选择（CoreAudio / WASAPI / ALSA / ASIO）
- 独占模式（Hog Mode）支持——JUCE 底层调用 CoreAudio 的 `kAudioDevicePropertyHogMode`
- 采样率自动切换
- 音频实时回调（`audioDeviceIOCallback`），约束与 CoreAudio render callback 一致

> 参考文档：[音频引擎方案对比.md](技术选型参考对比/音频引擎方案对比.md)

---

### 3.4 音频解码与元数据

**决策：JUCE AudioFormatManager**

JUCE 内置解码覆盖 6 种格式（FLAC / WAV / AIFF / MP3 / AAC / ALAC），全部为项目需求的格式。

| 格式 | 解码方案 | 说明 |
|------|---------|------|
| FLAC | JUCE AudioFormatManager | 内置，基于 libFLAC |
| WAV | JUCE AudioFormatManager | 内置 |
| AIFF | JUCE AudioFormatManager | 内置 |
| MP3 | JUCE AudioFormatManager | 内置 |
| AAC | JUCE AudioFormatManager | macOS 系统解码 |
| ALAC | JUCE AudioFormatManager | macOS 系统解码（CoreAudio AudioConverter） |
| 元数据 | JUCE AudioFormatReader::metadataValues | 常用 tag 够用，如需更完整支持可后续引入 TagLib |

> ⚠️ DSD 格式（DSF/DFF）暂不纳入。DSD 用户群体过于小众，待应用主体开发完毕后视情况再决定是否支持。

> 参考文档：[音频解码与元数据方案对比.md](技术选型参考对比/音频解码与元数据方案对比.md)

### 3.5 UI 框架

**决策：JUCE GUI（juce_gui_basics 模块）**

| 维度 | 评估 |
|------|------|
| 与音频核心协作 | C++ 零桥接。UI 控件直接调用音频核心对象，数据流清晰可追溯 |
| 跨平台 | 同一套 C++ UI 代码在 macOS / Windows / Linux 上渲染一致 |
| 风格统一 | 自绘控件 + LookAndFeel 定制，实现跨平台统一的简约现代风格 |
| 控件丰富 | TableListBox（播放列表）、TreeView（目录树）、Slider（音量）、FlexBox/Grid 现代布局 |
| 开发效率 | 不如 QML 声明式快，但消除了"UI 框架 ↔ 音频引擎"的桥接层，整体效率更高 |

**为什么不选 Qt？**
- Qt 的 UI（QML）和音频核心（C++）之间虽然都是 C++，但需要 QObject 封装和信号槽机制，相当于在 JUCE 的统一 C++ 之外又多了一层概念
- Qt 的 macOS 打包需要处理 LGPL 动态库分发
- 如果音频引擎选了 JUCE，UI 又选 Qt，等于引入两个大型框架，增大了学习负担

> 参考文档：[UI框架方案对比.md](技术选型参考对比/UI框架方案对比.md)

### 3.6 包管理与依赖管理

**决策：Git Submodule**

| 维度 | 评估 |
|------|------|
| 离线构建 | 所有依赖随仓库一并克隆，无需网络即可完成 CMake 构建 |
| 版本锁定 | 每个依赖的 commit SHA 精确记录在 Git 历史中，版本与代码同步，完全可追溯 |
| CI 友好 | GitHub Actions 上 `git submodule update --init` 后直接 `cmake -B build` |
| 适用性 | 本项目依赖 5 个第三方库（JUCE/spdlog/Catch2/nlohmann/SQLite），submodule 管理无负担 |
| 统一管理 | 全部依赖统一放在 `third_party/` 目录，`add_subdirectory()` 引入，机制一致 |

**注意**：JUCE 本身不作为 FetchContent 引入。JUCE 通过 git submodule 放在 `third_party/JUCE/`，通过 `add_subdirectory(JUCE)` 引入 CMake 构建。这是 JUCE 官方推荐的 CMake 集成方式。

> 参考文档：[包管理与依赖管理方案对比.md](技术选型参考对比/包管理与依赖管理方案对比.md)

### 3.7 日志框架

**决策：spdlog**

spdlog 是 C++ 日志的事实标准，独立于 JUCE。使用异步 logger（后台线程写日志），确保即使从音频实时线程调用也是无锁入队。

```cpp
// 初始化（应用启动时）
spdlog::init_thread_pool(8192, 1);  // 队列 8192 条，1 个后台线程
auto file_sink = std::make_shared<spdlog::sinks::rotating_file_sink_mt>(
    "logs/bitperfect.log", 1024 * 1024 * 10, 3);  // 10MB × 3 个文件轮转
auto logger = std::make_shared<spdlog::async_logger>(
    "bitperfect", file_sink, spdlog::thread_pool());
spdlog::set_default_logger(logger);

// 使用（任何线程，包括音频线程——异步 logger 不阻塞）
SPDLOG_INFO("音频设备已打开: {}", deviceName);
SPDLOG_DEBUG("缓冲区: {} frames, {}/{}Hz", frames, bitDepth, sampleRate);
```

**为什么不用 JUCE 的 Logger？** JUCE 的 `Logger` 类只支持同步写，不适合音频实时线程。spdlog 的异步 logger 是刚需。

> 参考文档：[日志框架方案对比.md](技术选型参考对比/日志框架方案对比.md)

### 3.8 测试框架

**决策：Catch2（v3）**

| 维度 | 评估 |
|------|------|
| 代码可读性 | `REQUIRE(a == b)` 自然语言风格，失败时自动打印左右值，符合项目"代码可读性优先"理念 |
| SECTION 机制 | 同一个测试用例内写多个子场景，共享 setup 代码但独立运行，适合"同一解码器、不同参数"的测试 |
| BENCHMARK | 验证音频解码性能、缓冲区处理延迟等非功能需求 |
| CMake 集成 | git submodule 引入，`add_test` + `ctest` 运行 |

### 3.9 多线程与并发

**决策：JUCE 内置线程方案**

JUCE 的线程模型天然匹配音频播放器的三类线程场景：

| 线程类型 | JUCE 方案 | 说明 |
|---------|----------|------|
| 音频实时线程 | `juce::AudioIODeviceCallback::audioDeviceIOCallback()` | 系统回调，硬实时约束，仅做 memcpy + 原子变量读取 + AbstractFifo 操作 |
| UI 主线程 | `juce::MessageManager::callAsync()` | 主线程消息循环，UI 控件更新入口 |
| I/O 工作线程 | `juce::ThreadPool` | 后台解码、文件扫描、数据库写入 |
| 线程间通信 | `juce::AbstractFifo` | 无锁环形缓冲区，专为音频场景设计 |

不再需要额外引入 moodycamel::ReaderWriterQueue——JUCE 的 AbstractFifo 就是为此场景设计的。

> 参考文档：[多线程与并发方案对比.md](技术选型参考对比/多线程与并发方案对比.md)

### 3.10 配置管理

**决策：JSON（nlohmann/json）**

| 维度 | 评估 |
|------|------|
| C++ 原生 | 直接在 C++ 层读写，不需要 ObjC 桥接 |
| 人类可读 | 调试时直接打开 `config.json` 查看，Git diff 友好 |
| API 易用 | `config["audio"]["sample_rate"]` 类型推导，`.value("key", defaultVal)` 默认值机制 |
| 原子写入 | 先写 `.tmp` 文件 → `rename()` 原子替换，防断电丢失 |

配置文件路径（跨平台）：
- macOS：`~/Library/Application Support/BitPerfect/config.json`
- Windows：`%APPDATA%/BitPerfect/config.json`
- Linux：`~/.config/BitPerfect/config.json`

> 参考文档：[配置管理方案对比.md](技术选型参考对比/配置管理方案对比.md)

### 3.12 音频可视化与 DSP

**决策：JUCE dsp::FFT**

| 功能 | 方案 | 说明 |
|------|------|------|
| 频谱分析（FFT） | **JUCE dsp::FFT** | 跨平台，API 现代。后续可针对 macOS 用 vDSP 优化 |
| 窗函数 | **JUCE dsp::WindowingFunction** | Hann / Hamming / Blackman 等内置 |
| 波形渲染 | **JUCE Graphics**（CoreGraphics 后端） | 在 `juce::Component::paint()` 中绘制，静态波形缓存为 `juce::Image` |
| VU 表 / 峰值表 | **自行实现 RMS + dB 转换** | 约 100 行 C++，放在音频回调中计算，通过 AbstractFifo 传给 UI |
| 均衡器（EQ） | 预留扩展点，暂不实现 | P1/P2 需求，架构上预留 `dsp::ProcessorBase` 处理链 |

> 参考文档：[音频可视化与DSP方案对比.md](技术选型参考对比/音频可视化与DSP方案对比.md)

### 3.13 崩溃报告与诊断

**决策：Sentry Native SDK（发布后集成）**

| 阶段 | 做法 |
|------|------|
| 开发阶段 | 不集成 Sentry，用 LLDB + Sanitizers（ASan / TSan / UBSan）调试 |
| 内测/发布阶段 | 集成 Sentry Native SDK，收集崩溃报告 + Breadcrumbs 轨迹 |

选择 Sentry 的理由：
- C++ 原生 SDK（`sentry-native`），零桥接
- Breadcrumbs 功能对诊断"特定 DAC + 特定采样率 + Hog Mode"组合触发的 crash 极其有用
- 免费额度 5000 events/月，个人项目够用
- 自托管选项（不想用 SaaS 可自建）

> 参考文档：[崩溃报告与诊断方案对比.md](技术选型参考对比/崩溃报告与诊断方案对比.md)

## 五、决策汇总表

| 序号 | 技术维度 | 选型 | 许可证 | 一句话理由 |
|:---:|---------|------|:---:|------|
| 1 | IDE | VS Code + Xcode 辅助 | 免费 | VS Code 跨平台主力编码，Xcode 负责 Instruments 性能分析和 macOS 打包 |
| 2 | 构建系统 | CMake | BSD | C++ 生态事实标准，JUCE 官方支持，VS Code 深度集成 |
| 3 | 音频引擎 | JUCE AudioDeviceManager | GPLv3 | 跨平台、文档最好、Hog Mode 支持完整 |
| 4 | 音频解码 | JUCE AudioFormatManager | GPLv3 | 6 种格式内置全覆盖（FLAC/WAV/AIFF/MP3/AAC/ALAC），DSD 暂不支持 |
| 5 | UI 框架 | JUCE GUI | GPLv3 | C++ 零桥接，跨平台自绘统一风格 |
| 6 | 包管理 | Git Submodule | — | 离线构建，版本精确锁定，全部依赖统一在 third_party/ 目录 |
| 7 | 日志 | spdlog | MIT | C++ 日志事实标准，异步安全，音频线程无阻塞 |
| 8 | 测试 | Catch2 v3 | BSL 1.0 | 可读性最好，SECTION 机制适合参数化测试 |
| 9 | 多线程 | JUCE 内置方案 | GPLv3 | 三类线程模型天然匹配，AbstractFifo 无锁通信 |
| 10 | 配置管理 | JSON（nlohmann/json） | MIT | C++ 原生读写，人类可读，跨平台 |
| 11 | 音乐库数据库 | SQLite | 公共域 | 全球最可靠嵌入式数据库，FTS5 全文搜索 |
| 12 | 音频可视化 | JUCE dsp::FFT | GPLv3 | 跨平台一致，后续可针对 macOS 用 vDSP 优化 |
| 13 | 崩溃报告 | Sentry Native SDK | BSL | C++ 原生 SDK，Breadcrumbs 诊断利器，免费额度够用 |

## 六、与原则的符合性检查

| 原则 | 符合情况 |
|------|---------|
| **跨平台优先** | ✅ JUCE + CMake + VS Code 全链路跨平台。macOS 先用，Windows/Linux 后续直接编译 |
| **低学习成本** | ✅ JUCE 文档和教程是音频开发领域最佳。相比「miniaudio + dr_libs + Qt + 自行集成」的组合方案，JUCE 的一站式方案减少了整合多个独立库的心智负担 |
| **统一跨平台 UI** | ✅ JUCE 自绘控件 + LookAndFeel 定制，跨平台一致外观 |
| **成熟活跃社区** | ✅ JUCE Forum 非常活跃，Ableton/Serum 等商业产品验证。spdlog/Catch2/SQLite 各自领域的工业标准 |
| **开源友好** | ✅ 所有选型许可证（GPLv3 / MIT / BSD / BSL / 公共域）均兼容开源分发 |
